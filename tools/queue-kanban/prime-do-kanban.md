# Prime: do-kanban

queue-kanban ŌĆö standalone Go module (`tools/queue-kanban/`, own `go.mod`) that walks the version-controlled `do-work/` Markdown tree and renders it as a Kanban board + completion calendar. Subcommands: `summary` | `generate --out DIR` | `serve`. It ships as part of the do-work skill and rides its version bumps; the ergonomic entry point is the `do-work board` action (`actions/board.md`), which builds and runs it for you.

## Read first
- `main.go` ŌĆö subcommand dispatch + flags (`--repo-root`, `--out`, `--port`; `--recent-window` on summary only)
- `walk.go` ŌĆö repo-root resolution (walks up for `do-work/`) + tree enumeration
- `model.go` ŌĆö Board model; column bucketing driven by REQ status
- `generate.go` ŌĆö static self-contained site; `//go:embed web/` + inline placeholders

## Do not edit
- `build/` outputs ŌĆö gitignored, regenerated by every build
- generated `index.html` / `board-data.js` ŌĆö products of `generate`, not source
- `go.sum` ŌĆö managed by the Go toolchain

## Must build
- Easiest: run `do-work board` (builds + serves for you; `do-work board static` for the shareable static artifact, `do-work board summary` for counts). See `actions/board.md`.
- Direct (portable, any repo with a `do-work/`): `cd <skill-dir>/tools/queue-kanban && go build -o queue-kanban .` ŌåÆ `./queue-kanban serve --repo-root <consumer-repo-root>` (live board) or `./queue-kanban generate --out DIR --repo-root <consumer-repo-root>` (static). The binary is gitignored in-place. Needs the Go toolchain (see `go.mod` for the required version).

## Traps
- **Columns track REQ status, not folder** ŌĆö `walk.go` keeps queue/working/archive only as provenance; `model.go` buckets by normalized `status`. A status-alias drift sends tickets to the wrong column.
- **Repo root is auto-discovered from CWD** ŌĆö prefer `--repo-root DIR` (what `actions/board.md` prescribes); the walk-up skips directories merely *named* `do-work` that are skill installs (SKILL.md at their top level, e.g. `.claude/skills/do-work/`) and errors non-zero only when no queue-holding ancestor exists.
- **`web/` is embedded, not read at runtime** ŌĆö edits need a rebuild; `serve` re-serves the SAME embedded assets `generate` inlines, never files on disk.
- **Separate Go module** ŌĆö build inside `tools/queue-kanban/`; a repo-root `go build ./...` from the consuming project won't reach it (it lives under `.claude/skills/do-work/`).
- **Loopback-only by default** ŌĆö `serve` binds `127.0.0.1:8090` (env `QUEUE_KANBAN_PORT`; 8090 avoids the main goserver :8080). Bare port values also bind loopback; LAN exposure requires an explicit `host:port`.

## Stakes
- `model.go` + `walk.go` ŌĆö status ŌåÆ column bucketing (`LoadBoard`)
  Req:   bucket every REQ into Pending / Claimed / Needs-input-or-Blocked / Recently-done by its normalized do-work `status`, not its tree location.
  Value: the board reflects the REAL do-work status vocabulary, so the operator and any manager it's shared with trust what each column means.
  Risk:  if status normalization drifts from `actions/work-reference.md`'s Schema Read Contract, tickets silently land in the wrong column or vanish from the board. Reversible (re-run after a fix) but actively misleads viewers until caught.
- `generate.go` + `serve.go` ŌĆö shared `//go:embed web/` frontend contract
  Req:   `generate` (static) and `serve` (live) render from the SAME embedded `web/` assets; the static output must run from `file://` with zero network via the `board-data.js` sibling.
  Value: one hand-authored frontend source powers two outputs ŌĆö a manager-shareable static artifact and a live local board.
  Risk:  editing `web/` without rebuilding ships stale UI; breaking the offline `board-data.js` contract makes the shared static artifact render blank for anyone opening it from disk.

- REQ-015: synthetic test tickets must set `OriginalStatus` for unrecognized-status warning assertions to exercise the real path; `deferred` removed from the recognized set
- REQ-016: the `severity` frontmatter vertical was dead (no producer) and is fully removed; the similar-looking `batch` field HAS real producers ŌĆö re-check before any same-shape cleanup
- REQ-017: bind before announcing ŌĆö `serve` must `net.Listen` first, then print the banner / open the browser, then `Serve(listener)`; the old `ListenAndServe` ordering printed a false "live board at ŌĆ”" line on port collisions. For opener-style side effects, prefer parameter-injected seams (`browserOpener func`, `goos` as an argument) over mutable package vars ŌĆö testable without launching anything
- 0.107.0: `cancelled` (won't-do, written by `do-work abandon`) is terminally *resolved* but not terminally *successful* ŌĆö it shares Recently-done and the calendar via `isTerminalResolvedStatus`, while `isCompletedStatus` stays the success-only gate. Keep the two classifiers distinct; success-readers must never widen to the resolved set
- 0.125.0: `reserved` (allocated to another worktree/cloud session, written by `do-work reserve`) shares the Claimed column but is grayed out client-side via `[data-status="reserved"]`; `reservationStaleAfter` (24h) mirrors actions/work.md Step 1's stale-reservation threshold ŌĆö change them together. Staleness warns and suggests recategorizing; the board never auto-releases
- 0.129.0: bare `blocked` (waiting on an external condition, written at capture or by work.md's mid-run blocked flip) shares the Needs-input/Blocked column via `isNeedsInputOrBlockedStatus` ŌĆö distinct from the two `blocked-*` holding states and from `pending-answers`. `blocked_by`/`blocked_at`/`blocked_check` are display-only (the `[data-status="blocked"]` "blocked by" badge + drawer rows); the board never runs `blocked_check` ŌĆö the work pipeline does. `blocked_by` parses as a list (free-text scalar ŌåÆ one element; legacy id-lists render joined) but is NOT a dependency edge
