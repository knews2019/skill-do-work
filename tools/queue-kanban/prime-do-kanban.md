# Prime: do-kanban

queue-kanban ‚Äî standalone Go module (`tools/queue-kanban/`, own `go.mod`) that walks the version-controlled `do-work/` Markdown tree and renders it as a Kanban board + completion calendar. Subcommands: `summary` | `generate --out DIR` | `serve`. It ships as part of the do-work skill and rides its version bumps; the ergonomic entry point is the `do-work board` action (`actions/board.md`), which builds and runs it for you.

## Read first
- `main.go` ‚Äî subcommand dispatch + flags (`--repo-root`, `--recent-window`, `--out`, `--port`)
- `walk.go` ‚Äî repo-root resolution (walks up for `do-work/`) + tree enumeration
- `model.go` ‚Äî Board model; column bucketing driven by REQ status
- `generate.go` ‚Äî static self-contained site; `//go:embed web/` + inline placeholders

## Do not edit
- `build/` outputs ‚Äî gitignored, regenerated by every build
- generated `index.html` / `board-data.js` ‚Äî products of `generate`, not source
- `go.sum` ‚Äî managed by the Go toolchain

## Must build
- Easiest: run `do-work board` (builds + serves for you; `do-work board static` for the shareable static artifact, `do-work board summary` for counts). See `actions/board.md`.
- Direct (portable, any repo with a `do-work/`): `cd <skill-dir>/tools/queue-kanban && go build -o queue-kanban .` ‚Üí `./queue-kanban serve` (live board) or `./queue-kanban generate --out DIR` (static). The binary is gitignored in-place. Needs the Go toolchain (see `go.mod` for the required version).

## Traps
- **Columns track REQ status, not folder** ‚Äî `walk.go` keeps queue/working/archive only as provenance; `model.go` buckets by normalized `status`. A status-alias drift sends tickets to the wrong column.
- **Repo root is auto-discovered from CWD** ‚Äî run from inside the repo or pass `--repo-root DIR`; from outside a `do-work/` ancestor it exits non-zero.
- **`web/` is embedded, not read at runtime** ‚Äî edits need a rebuild; `serve` re-serves the SAME embedded assets `generate` inlines, never files on disk.
- **Separate Go module** ‚Äî build inside `tools/queue-kanban/`; a repo-root `go build ./...` from the consuming project won't reach it (it lives under `.claude/skills/do-work/`).
- **Port :8090, not :8080** ‚Äî `serve` defaults to :8090 (env `QUEUE_KANBAN_PORT`) to avoid the main goserver.

## Stakes
- `model.go` + `walk.go` ‚Äî status ‚Üí column bucketing (`LoadBoard`)
  Req:   bucket every REQ into Pending / Claimed / Needs-input-or-Blocked / Recently-done by its normalized do-work `status`, not its tree location.
  Value: the board reflects the REAL do-work status vocabulary, so the operator and any manager it's shared with trust what each column means.
  Risk:  if status normalization drifts from `actions/work-reference.md`'s Schema Read Contract, tickets silently land in the wrong column or vanish from the board. Reversible (re-run after a fix) but actively misleads viewers until caught.
- `generate.go` + `serve.go` ‚Äî shared `//go:embed web/` frontend contract
  Req:   `generate` (static) and `serve` (live) render from the SAME embedded `web/` assets; the static output must run from `file://` with zero network via the `board-data.js` sibling.
  Value: one hand-authored frontend source powers two outputs ‚Äî a manager-shareable static artifact and a live local board.
  Risk:  editing `web/` without rebuilding ships stale UI; breaking the offline `board-data.js` contract makes the shared static artifact render blank for anyone opening it from disk.
