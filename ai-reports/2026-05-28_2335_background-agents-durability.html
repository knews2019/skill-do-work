Background-agent work, made survivable

Disk is the source of truth — not the chat transcript. Sub-agents write their findings to files; the orchestrator synthesizes from those files. When a fan-out session is interrupted, compacted, or hits a provider error, the work is still on disk and a fresh session can finish it.

02The durability pattern

Four moves. The same four whether you have a native orchestration engine or a single sequential context — only the machinery you hand-roll changes.

Create the run directory before any spawn

Make do-work/runs/<action>-<YYYY-MM-DD-HHMM>/ first, with a shell-derived timestamp so reruns and recovery can find it. Nothing is spawned before it exists.

Each sub-agent writes its own file; returns one line

The agent writes its full findings to its assigned path and returns only a one-line status to the orchestrator — never the findings inline. Small assembly turns stay cheap and shrink the corruption window.

Write a manifest; spawn in bounded waves

A manifest.md records each agent, its slice, its expected filename, and its landed status. Spawn in waves sized to the concurrency limit — not one unbounded fan-out — updating the manifest as each wave lands.

Synthesize from the files on disk

When the waves are done, read the findings files and assemble from them — never from what agents "said" in chat. This is what makes synthesis behave identically in the original session and in a fresh recovery session that never saw the spawns.

05The failure it can't prevent

Be honest about the ceiling. This pattern does not stop the corruption below — that fault is harness/API-level, beyond a markdown rule's reach. It makes it recoverable.

⚠ Reasoning-block corruption

On harnesses with a signed thinking block, an assistant turn that holds an open signed thinking block and long-running background spawns can be corrupted if it's interrupted or re-stitched while still open. Once the malformed turn is written to the session log, every resume replays it and re-throws HTTP 400 — "thinking blocks cannot be modified".

The session becomes permanently un-resumable. The corruption is per-transcript — other sessions on the machine are unaffected. The one-line-status rule shrinks the window but is mitigation, not prevention.

Recovery — fresh session, re-spawn only what's missing

Do NOT resume the poisoned conversation. Resuming replays the corrupt turn and re-throws the error every time.
Start a fresh session and re-invoke the same action.
Let the action detect the existing run directory (do-work/runs/<action>-*) and read the manifest.
Re-spawn only the agents whose output files are missing. Agents that already wrote their file are done.
Synthesize from disk as normal — identical to the original run.
Delete the poisoned transcript once recovery succeeds.

06Match the pattern to the harness

One invariant carries down all three rungs: disk is the source of truth. Use the highest rung the harness supports.

①

Native orchestration engine

A deterministic fan-out primitive with journaled resume — caps concurrency, returns structured per-agent output, replays cached results on re-run. Prefer it: you get bounded waves, structured hand-off, and manifest-plus-re-spawn recovery for free, usually detached from the orchestrator turn. Still write each slice to disk — journal and files are belt-and-suspenders.

best — least hand-rolling
②

Manual parallel / background spawns

Can spawn concurrent sub-agents but offers no orchestration engine. Hand-roll the pattern exactly as steps 1–4 describe: run directory, per-slice findings files, manifest, bounded waves, synthesize from disk.

hand-roll steps 1–4
③

Sequential in-context

No parallel or background support at all — but don't skip the pattern. Run the slices one at a time in the current context, still create the run dir, still write each slice's file as you complete it, still update the manifest, still synthesize from disk. A sequential run that crashes halfway is recoverable because completed slices are already on disk.

still durable
