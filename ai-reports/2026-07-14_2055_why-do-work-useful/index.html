Why do-work is useful

Auto-ticketing that executes, tests, reviews, and presents the work.

Every request becomes a ticket, and the tickets close themselves — that is the claim, and this page walks the mechanism behind each part of it: what file, what field, what action actually delivers it. If you already drive an agent all day but work ad-hoc — one long thread, no tickets — this is not a pitch; everything here is checkable against the skill's own files.

Mechanism 1 — Durable intent

Requests become files, so work survives context loss

Every capture writes two things to disk: a UR folder holding your verbatim input as the permanent source of truth, and one or more REQ files — plain Markdown with YAML frontmatter — that enter do-work/queue/. The frontmatter carries a machine-readable status, so the request's position is a fact on disk, not a memory of the conversation.

Because the state lives in do-work/ rather than the transcript, it doesn't matter if the context window fills, the terminal crashes, or you hand the thread to a different model tomorrow. The next agent reads status: pending and knows exactly what's left. Even a multi-step run persists its position in do-work/pipeline.json and resumes from the last completed step.

Mechanism 2 — Separation of capture and execution

You can bank an idea mid-task without derailing

The capture-requests action writes the UR and REQ files and then stops. It is designed to never "helpfully" slide into building — SKILL.md states the rule plainly: capture captures, work executes, and they are strictly separate. So dropping a new request into the queue while you're deep in something else costs you nothing: the idea is banked, the current work is untouched.

Capture is also where your attention is worth the most. Clarifying questions — each one offering concrete options, not open-ended "what do you mean?" — happen while you're present. Compound inputs ("the search is slow, also add an export button, and fix the header") split into separate REQs automatically. Then the build phase runs later, on its own, without blocking on you.

The REQ lifecycle

    completed --> [*]
    completed_with_issues --> [*]
    cancelled --> [*]

One field, one lifecycle

The status field is the whole state machine. work claims a pending REQ, builds it, and stamps a terminal status. Ambiguity doesn't stall the loop — the builder marks it pending-answers and moves on, and do-work clarify collects those questions as a batch for you to answer later. A won't-do decision becomes cancelled (terminal, recorded, not a silent delete). A failure spawns a follow-up REQ so nothing evaporates.

Mechanism 3 — Built-in quality gates

Review and guardrails are structural, not optional politeness

After a REQ is built, review-work runs a requirements check, a code review, and acceptance testing against the original intent — not a vibe-check that it compiled. Each action file ends with a Verification Checklist that demands evidence, not a self-declared pass.

The guardrails load by trigger condition, just-in-time from crew-members/. karpathy.md is always active during implementation — think before coding, surgical changes, verifiable success criteria. anti-slop.md loads before any human-facing artifact and enforces eight principles (lead with the conclusion, verify every claim, compress, disclose what's AI-generated). This very report was written under those rules.

Loaded when its trigger fires

  • karpathy.md — always, during build
  • anti-slop.md — before human-facing artifacts
  • prompt-injection.md — before ingesting outside content
  • clear-questions.md — before any interactive question
  • maintenance.md — on a maintenance: true REQ

Mechanism 4 — Observability

The queue is inspectable at a glance

Because work is files with a known schema, tools can read it. do-work board builds the shipped Go tool in tools/queue-kanban/ and renders the queue as a live Kanban board at :8090 (or a static HTML snapshot, or a column-count summary). do-work forensics is a read-only diagnostic that flags stuck work, hollow completions, and orphaned URs. do-work roadmap classifies pending REQs as ready, needs-clarification, blocked, or stale. do-work inspect explains what's uncommitted and whether it's ready.

Read-only lenses on the same queue

  • board — Kanban view of every status column
  • roadmap — ready / blocked / stale + TDD posture
  • forensics — stuck, hollow, orphaned work
  • inspect — what changed, and is it shippable

Mechanism 5 — Knowledge compounding

What one REQ learned becomes context for the next

A completed REQ can carry a ## Lessons Learned section — the surprising thing the build revealed. Once review passes, do-work offers to promote those lessons into the project's knowledge base via the kb-lessons-handoff. The REQ's own frontmatter tracks the outcome in kb_status — promoted, pending, declined, or skipped — so the handoff is auditable and never blocks archival.

The effect over time: insight stops dying with the thread that produced it. An ad-hoc chat forgets what it discovered the moment it scrolls off. A queue with a KB handoff turns each finished task into reusable ground for the tasks that come after it.

The value loop

Idea to queue to pipeline to archive to knowledge base

Each stage writes a durable artifact the next stage reads. The knowledge base feeds back into future work, which is where the loop compounds.

Where the value concentrates

The trail of intent is the point; the code is the side effect

Read together, the five mechanisms produce one artifact an ad-hoc chat can't: a living record that traces from the words you typed, through every builder decision and scope call, to the commit that shipped. The UR preserves the verbatim ask, the REQ preserves the validated intent, and the appended sections preserve how that intent got realized.

None of this requires Claude Code specifically — the action files are standalone prompts that run in any agent that can read files and run shell commands. The skill doesn't replace the agent's judgment; it raises the floor, so the same good steps happen every time instead of only when the agent happens to feel careful.
